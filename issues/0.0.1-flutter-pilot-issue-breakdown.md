## 1. Define Runtime Adapter Contract

Type: HITL

Blocked by: None - can start immediately

User stories covered: 39, 40, 41, 42

## What to build

Define Flutter Pilot's narrow Runtime Adapter contract above `mcp_flutter`. The adapter must cover Runtime Target connection, Finder resolution, tap, type, scroll, waitFor, screenshot capture, Snapshot capture, Widget Tree capture, and Logs capture. Runtime errors are collected as part of Logs in the first version.

This slice also adds a fake adapter for runner tests and performs one manual `mcp_flutter` calibration pass. Prefer an available `mcp_flutter` API/SDK call path; use CLI subprocess execution only as a fallback when the API path is unavailable or insufficient. Calibration must cover discovery, screenshot, semantic snapshot, Logs, Finder byText/byKey/byType, multiple Finder Match descriptions, WidgetBounds availability, and scroll without Finder.

## Acceptance criteria

- [ ] Adapter contract covers first-version actions and capture artifacts.
- [ ] Fake adapter can simulate success, failure, zero Finder Matches, one Finder Match, and multiple Finder Matches.
- [ ] Manual `mcp_flutter` calibration checks API availability, CLI fallback needs, discovery, screenshot, semantic snapshot, Logs, Finder byText/byKey/byType, multiple Finder Match descriptions, WidgetBounds availability, and scroll without Finder.
- [ ] Contract tests verify the adapter input and output shapes.

## 2. Replay Successful Scenario And Write Basic Run Report

Type: AFK

Blocked by: Issue 1

User stories covered: 1, 2, 3, 4, 5, 6, 7, 8, 9, 24, 36, 41

## What to build

Make `run` execute a valid Scenario instead of stopping at command-shell validation. The runner should execute Steps in order through the Runtime Adapter, record step status, duration, and action result, then write a basic `run_report.json` for a fully successful run.

## Acceptance criteria

- [ ] `run` executes successful tap, type, scroll, and waitFor Steps through the adapter.
- [ ] A fully successful run exits with code `0`.
- [ ] The run report includes Scenario metadata and ordered Step results.
- [ ] Runner tests use the fake adapter and do not require a live Flutter app.

## 3. Enforce Finder Match Cardinality

Type: AFK

Blocked by: Issue 2

User stories covered: 2, 3, 4, 5, 6, 36, 41

## What to build

Before executing a Finder-based action, apply the first-version Finder Match rule: zero matches fail the Step, one match executes the action, and multiple matches fail the Step.

## Acceptance criteria

- [ ] Zero Finder Matches fail the Step with a clear reason.
- [ ] One Finder Match executes the action.
- [ ] Multiple Finder Matches fail the Step with a clear reason.
- [ ] The run report records the failed Step and Finder Match failure reason.

## 4. Implement WaitFor Timing Behavior

Type: AFK

Blocked by: Issue 2, Issue 3

User stories covered: 9, 36, 41

## What to build

Implement `waitFor` polling behavior. The action should keep checking the Finder until exactly one Finder Match appears, the timeout expires, or multiple matches appear.

## Acceptance criteria

- [ ] `waitFor` succeeds when exactly one Finder Match appears before timeout.
- [ ] `waitFor` fails when timeout expires with zero matches.
- [ ] `waitFor` fails when multiple matches appear.
- [ ] Omitted `timeoutMs` uses the first-version default of `3000`.

## 5. Create Stable Run Directories And Artifact Metadata

Type: AFK

Blocked by: Issue 2

User stories covered: 16, 17, 23, 24, 35, 37, 43

## What to build

Introduce the Artifact Store. Each run should create a stable run directory, persist a copy of the Scenario, store Step metadata, and expose artifact paths through the run report.

## Acceptance criteria

- [ ] Run directories use timestamp plus Scenario name.
- [ ] The run report references artifact paths instead of embedding large artifacts.
- [ ] Artifact names include Step number and Step Label when available.
- [ ] Tests verify run directory layout and metadata references.

## 6. Capture Screenshots And Snapshots

Type: AFK

Blocked by: Issue 1, Issue 5

User stories covered: 15, 16, 17, 24, 35, 39

## What to build

Make `capture` Steps collect screenshots and Snapshots through the Runtime Adapter and write them through the Artifact Store. The run report should reference those artifacts from the producing Step.

## Acceptance criteria

- [ ] `capture: {}` collects screenshot and Snapshot artifacts by default.
- [ ] Screenshot and Snapshot artifacts are associated with Step number and Step Label.
- [ ] The run report records screenshot and Snapshot paths.
- [ ] Tests use the fake adapter to verify capture behavior.

## 7. Capture Logs

User stories covered: 19, 20, 23, 24, 35, 39

Extend `capture` Steps to collect Logs. Runtime errors should be included in Logs when `mcp_flutter` exposes them. This must follow the default `capture: {}` bundle while allowing explicit overrides.

- [ ] Scenario authors can explicitly disable Logs capture.
- [ ] The run report records Logs artifact paths.
- [ ] Tests cover default capture behavior and explicit overrides.

Type: AFK

Blocked by: Issue 3, Issue 5, Issue 6, Issue 7

## What to build

When a Step fails, automatically collect the default diagnostic bundle even if the Scenario did not include an explicit capture Step at that point.

- [ ] Widget Tree capture remains disabled by default.
- [ ] The run report marks failure artifacts clearly.

## 9. Implement Real `--until` Stop Points

Type: AFK

Blocked by: Issue 2

User stories covered: 10, 11, 12, 34

## What to build

Turn `--until` from CLI validation into runner behavior. The runner should execute through the selected 1-based Step number or Step Label and stop after that Step completes.

## Acceptance criteria

- [ ] `--until 1` executes Step 1 and then stops.
- [ ] `--until <label>` executes through the Step with that label and then stops.
- [ ] Steps that were not executed have a clear report status.
- [ ] Invalid `--until` values remain CLI usage errors.

## 10. Implement `--print` Diagnostics After `--until`

Type: AFK

Blocked by: Issue 6, Issue 7, Issue 9

User stories covered: 13, 14, 18, 32, 34

## What to build

After the target Step selected by `--until` completes, print the requested diagnostic output: `snapshot`, `widget-tree`, or `logs`. Screenshots remain file artifacts and are not printed to stdout.

## Acceptance criteria

- [ ] `--print snapshot` prints structured Snapshot output.
- [ ] `--print widget-tree` prints Widget Tree output.
- [ ] `--print logs` prints current Logs, including runtime errors when available.
- [ ] `--print` does not support screenshot stdout output.

## 11. Add Diagnostic Reducer

Type: AFK

Blocked by: Issue 6, Issue 7, Issue 10

User stories covered: 18, 32, 34, 35

## What to build

Convert raw Snapshot, Widget Tree, and Logs data into compact agent-friendly summaries. The summary should preserve visible text, interactive widgets, route-like context, runtime failures from Logs, and likely suspects while removing low-value noise.

## Acceptance criteria

- [ ] Reducer output preserves visible text and interactive widgets.
- [ ] Reducer output preserves runtime failures from Logs and route-like context.
- [ ] Reducer output removes noisy implementation details that are not useful for agent diagnosis.
- [ ] Tests use representative Snapshot and Widget Tree fixtures.

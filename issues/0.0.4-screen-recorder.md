# 0.0.4 Screen Recorder Issue Breakdown

## 1. Scaffold `screen_recorder` Core API With Fake Backend

GitHub: https://github.com/drown0315/flutter_pilot/issues/58

## Parent

https://github.com/drown0315/flutter_pilot/issues/57

## What to build

Create the independent `screen_recorder` package foundation with a programmatic
core API and a fake recording backend. This slice should make Recording Device,
Recording Session, start/stop/discard behavior, output naming, result metadata,
and stable error codes usable without any real Android or iOS device.

The completed slice should be demoable through tests that start and stop a fake
Recording Session and receive a final saved video path.

## Acceptance criteria

- [ ] The repository contains an independent `screen_recorder` Dart package with a public library API.
- [ ] The public API exposes `listDevices`, `startRecord`, `stopRecord`, and discard behavior.
- [ ] `startRecord` returns a Recording Session and `stopRecord` returns a Recording Result with final path, timestamps, duration, file size, and MIME type.
- [ ] Output configuration uses output directory plus output name without extension, supports generated names, rejects paths/extensions in output names, and fails on existing outputs unless overwrite is explicit.
- [ ] The package uses one `ScreenRecorderException` type with stable error codes for core failure cases.
- [ ] The same recorder can hold parallel sessions for different Recording Devices and rejects a second active session for the same device.
- [ ] Unit tests cover fake backend list/start/stop/discard behavior, output naming, result metadata, stable error codes, and same-device already-recording behavior.

## Blocked by

None - can start immediately

## 2. Add Android Recording Device Support Through ADB

GitHub: https://github.com/drown0315/flutter_pilot/issues/59

## Parent

https://github.com/drown0315/flutter_pilot/issues/57

## What to build

Add Android Recording Device support behind the `screen_recorder` API. The slice
should discover Android devices through Android Debug Bridge, resolve selectors
using the agreed exact and prefix matching behavior, start Android screen
recording to a device-side temporary MP4, stop by ending the recording process,
pull the final file locally, and clean up the device-side temporary file.

The completed slice should make Android usable end-to-end through the core API
with fake process tests.

## Acceptance criteria

- [ ] Android Recording Devices are discovered from ADB output and exposed through `listDevices`.
- [ ] Android selector matching supports exact id, exact name, and case-insensitive name prefix matching.
- [ ] `startRecord` starts Android `screenrecord` against a device-side temporary `.mp4` path.
- [ ] `stopRecord` stops the recording process, pulls the device-side file to the local final `.mp4`, removes the device-side temporary file, and returns a Recording Result.
- [ ] Discard stops recording and removes the device-side temporary file without returning a saved recording.
- [ ] Missing ADB, command failures, stop failures, and cleanup-relevant raw output are surfaced through `ScreenRecorderException` error codes and diagnostics.
- [ ] Tests use fake process execution and do not require a real Android device.

## 3. Add iOS Simulator Recording Device Support Through `simctl`

GitHub: https://github.com/drown0315/flutter_pilot/issues/60

## Parent

https://github.com/drown0315/flutter_pilot/issues/57

## What to build

Add iOS Simulator Recording Device support behind the `screen_recorder` API. The
slice should discover booted or available simulator devices through
`xcrun simctl`, resolve selectors using exact and prefix matching, start
`simctl io recordVideo` as a managed long-running process, and stop or discard
the local `.mov` recording through the common Recording Session lifecycle.

The completed slice should make simulator recording usable end-to-end through
the core API with fake process tests.

## Acceptance criteria

- [ ] iOS Simulator Recording Devices are discovered from `simctl` output and exposed through `listDevices`.
- [ ] Simulator selector matching supports exact UDID, exact name, and case-insensitive name prefix matching.
- [ ] `startRecord` starts `xcrun simctl io <device> recordVideo <output>` and records to a backend-native `.mov` path.
- [ ] `stopRecord` signals the long-running process, waits for finalization, verifies a non-empty local `.mov`, and returns a Recording Result.
- [ ] Discard stops the process and removes the local output file.
- [ ] Missing `xcrun`/`simctl`, command failures, and stop failures are surfaced through `ScreenRecorderException` error codes and diagnostics.
- [ ] Tests use fake process execution and do not require a real simulator.

## 4. Add Physical iOS Recording Device Support With Native Helper

GitHub: https://github.com/drown0315/flutter_pilot/issues/61

## Parent

https://github.com/drown0315/flutter_pilot/issues/57

## What to build

Add physical iOS Recording Device support behind the `screen_recorder` API using
an in-package Swift helper based on the validated AVFoundation/CoreMediaIO
capture approach. The slice should build and invoke the helper from the Dart
package, parse native capture-device discovery output, start recording one
selected physical iOS capture device to `.mov`, stop the helper so it finalizes
the movie file, and discard by cleaning up the local output.

The implementation must not shell out to the local `ios_screen` prototype or
depend on files outside the `screen_recorder` package.

## Acceptance criteria

- [ ] The package contains its own native helper source for listing and recording physical iOS screen capture devices.
- [ ] Physical iOS Recording Devices are discovered through helper output and exposed through `listDevices`.
- [ ] Physical iOS selector matching supports exact native identity/name and case-insensitive name prefix matching.
- [ ] `startRecord` builds or locates the helper, starts it as a managed long-running process, and records to a backend-native `.mov` path.
- [ ] `stopRecord` signals the helper, waits for movie finalization, verifies a non-empty local `.mov`, and returns a Recording Result.
- [ ] Discard stops the helper and removes the local output file.
- [ ] Missing Swift toolchain, macOS permission/device failures, helper failures, and stop failures are surfaced through `ScreenRecorderException` error codes and diagnostics.
- [ ] Tests use fake process execution and representative helper output; they do not require a physical iPhone.

## 5. Wire Default Multi-Backend Recording Device Resolution Priority

GitHub: https://github.com/drown0315/flutter_pilot/issues/62

## Parent

https://github.com/drown0315/flutter_pilot/issues/57

## What to build

Connect the Android, iOS Simulator, and physical iOS backends behind the default
Recording Device resolver. The resolver should search in the agreed priority
order, stop at the first backend match, support exact and case-insensitive
name-prefix matching, and allow callers to narrow discovery with a platform
filter.

The completed slice should make default `listDevices` and selector-based
`startRecord` behavior deterministic across all supported backend families.

## Acceptance criteria

- [ ] The default resolver searches Android, then iOS Simulator, then physical iOS backends.
- [ ] Resolution stops as soon as a backend matches the selector.
- [ ] Exact id/native identity, exact name, and case-insensitive name-prefix matching are applied consistently across supported backend families.
- [ ] Platform filtering can narrow resolution to Android, iOS Simulator, or physical iOS.
- [ ] Device-not-found and ambiguous selection cases are surfaced with stable `ScreenRecorderException` error codes.
- [ ] Tests prove backend priority, short-circuit behavior, prefix matching, and platform filtering with fake backend data.

## 6. Add Thin Interactive CLI For Recording Sessions

GitHub: https://github.com/drown0315/flutter_pilot/issues/63

## Parent

https://github.com/drown0315/flutter_pilot/issues/57

## What to build

Add a thin `screen_recorder` CLI that uses the core library API for interactive
foreground Recording Sessions. The CLI should start a recording for a selected
Recording Device, keep the process in the foreground, map `s` to stop and save,
map `q` to discard, and print the final saved file path when the user saves.

The CLI must remain a shell around the core API, not a second implementation of
recording behavior.

## Acceptance criteria

- [ ] The package exposes a `screen_recorder` executable.
- [ ] The CLI can start a foreground recording for a device selector and optional output directory/name.
- [ ] Pressing `s` calls `stopRecord` and prints `Saved recording: <final-path>`.
- [ ] Pressing `q` calls discard behavior and exits without reporting a saved recording.
- [ ] CLI argument validation and error rendering use `ScreenRecorderException` codes from the core API.
- [ ] CLI behavior is tested through subprocess or fakeable interaction tests without requiring real devices.

## Blocked by

- https://github.com/drown0315/flutter_pilot/issues/62

- [ ] Documentation explains output directory/name behavior and backend-native `.mp4`/`.mov` formats.
- [ ] Documentation lists Android, iOS Simulator, and physical iOS prerequisites.
- [ ] Documentation includes manual smoke commands or checklist items for Android, iOS Simulator, and physical iOS.
- [ ] Documentation states that Flutter Pilot integration, Scenario YAML video actions, and VM service discovery are out of scope for this version.

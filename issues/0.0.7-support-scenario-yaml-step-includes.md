# 0.0.7 Support Scenario YAML Step Includes

Parent: https://github.com/drown0315/flutter_pilot/issues/78

## 1. Scenario Step Library Parser

## Parent

https://github.com/drown0315/flutter_pilot/issues/78

## What to build

Add file-based Step Include support to Scenario parsing. An Entry Scenario should be able to reference Step Library YAML files from its `steps` list, and `ScenarioParser.parseFile(...)` should expand those libraries into a flat Scenario before execution. The parser should support relative and absolute local include paths, nested includes, cycle detection, strict Step Library schema rules, duplicate Step Label validation after expansion, and Step Source metadata for expanded Steps.

This slice should keep include semantics inside parsing. A Step Include is not a Step, not a StepAction, and should not require Runner changes to execute.

## Acceptance criteria

- [x] `ScenarioParser.parseFile(...)` accepts `steps` entries shaped as pure Step Includes with a string `include` path.
- [x] Included Step Library files contribute their Steps at the include location and the returned Scenario has a flat, re-indexed Step list.
- [x] Relative include paths resolve relative to the YAML file that contains the include, and absolute include paths are accepted.
- [x] Nested includes are supported without a fixed maximum depth.
- [x] Include cycles are detected using canonicalized file identity where available and fail with a validation error instead of recursion overflow.
- [x] Step Libraries reject top-level fields other than `steps`, including `scenario`.
- [x] Include entries reject extra fields such as `label`, action keys, or future-looking metadata.
- [x] Duplicate Step Labels fail after expansion across the entire Entry Scenario.
- [x] Missing files and invalid included YAML produce validation errors that identify the include path.
- [x] In-memory parsing without file context does not silently resolve includes from the process working directory.
- [x] Parser tests cover successful expansion, nested relative paths, absolute paths, missing files, invalid YAML, cycles, duplicate labels, Step Library `scenario` rejection, and include-entry extra-field rejection.

Progress:

- 2026-06-30: Parser slice implemented with focused parser coverage. Verified with `dart test test/scenario_parser_test.dart`.

## Blocked by

None - can start immediately

## 2. Scenario Step Library CLI Validation

## Parent

https://github.com/drown0315/flutter_pilot/issues/78

## What to build

Expose Step Include behavior through the existing CLI validation and run entrypoints. `flutter_pilot validate` should validate include chains without a Runtime Target, and `flutter_pilot run` should accept include-backed Entry Scenarios as ordinary Scenarios after parsing. CLI output should preserve the existing validation style while making include-related failures locatable from the Entry Scenario.

This slice should verify external CLI behavior through subprocess tests and should not add a separate Step Library validation mode.

## Acceptance criteria

- [ ] `flutter_pilot validate <scenario.yaml>` succeeds for an Entry Scenario that includes valid Step Libraries.
- [ ] `flutter_pilot validate <scenario.yaml>` fails for missing include files, invalid included YAML, invalid included Steps, Step Library `scenario` metadata, duplicate labels after expansion, and include cycles.
- [ ] Validation errors include paths that identify the include location and display path.
- [ ] `flutter_pilot validate --json` includes include-related validation paths in the same JSON error shape as other Scenario validation errors.
- [ ] `flutter_pilot run <scenario.yaml> --target <runtime-target>` accepts include-backed Entry Scenarios after parsing and applies existing CLI argument validation to the expanded Scenario.
- [ ] `--until` by Step number and Step Label uses the expanded flat Step list.
- [ ] A YAML file with only `steps` remains valid when passed directly as an Entry Scenario, with existing Scenario name fallback behavior preserved.
- [ ] CLI subprocess tests cover successful include-backed validation, JSON validation failure output, `run --until` against expanded labels or indexes, and rejection of invalid include chains.

## Blocked by

- Scenario Step Library Parser

## 3. Expanded Scenario Artifacts

## Parent

https://github.com/drown0315/flutter_pilot/issues/78

## What to build

Ensure run artifacts represent the expanded Scenario that actually executed. The run artifact containing the Scenario definition should include the flat Step list produced by include expansion and should preserve Step Source metadata where appropriate. Execution result reports should continue to focus on Step outcomes and should not require downstream tools to understand a hierarchical include tree.

This slice should keep Runner semantics unchanged: the Runner receives and executes ordinary expanded Steps.

## Acceptance criteria

- [ ] The Scenario artifact written for a run contains the expanded flat Step list, not raw Step Include directives.
- [ ] Expanded Step indexes in artifacts match the Step indexes used by the Runner and `--until`.
- [ ] Step Source metadata is available in the Scenario artifact for Steps parsed from files.
- [ ] Root Entry Scenario Steps and included Step Library Steps both have source metadata when parsed through `parseFile(...)`.
- [ ] Execution result reports continue to expose the existing flat Step outcome shape.
- [ ] HTML report generation and Run Diff behavior continue to work with include-backed Scenario Runs without needing include-tree awareness.
- [ ] Artifact store or runner tests verify artifact output for include-backed Scenarios.

## 4. Include Chain Ready For Progress

## Parent

https://github.com/drown0315/flutter_pilot/issues/78

## What to build

Stabilize the Step Source and Include Chain contract so future CLI progress output can fold included Step groups without changing parsing or execution semantics. This slice does not implement progress output. It ensures each expanded Step carries enough origin data to identify the current source file, display path, YAML path, and full Include Chain from the Entry Scenario.

The contract should support future display such as showing a group for `steps/login.yaml` while the actual Runner and reports still operate on individual expanded Steps.

## Acceptance criteria

- [ ] Step Source records normalized file identity for tooling and display path for user-facing output.
- [ ] Step Source records the YAML path of the Step inside its source file.
- [ ] Include Chain records the complete path from Entry Scenario through nested Step Includes to the Step's source file.
- [ ] Display paths preserve the user-written include path where practical.
- [ ] Source metadata is optional for in-memory parsed Scenarios that do not have file context.
- [ ] Tests verify source metadata for root Steps, directly included Steps, nested included Steps, repeated includes from different branches, and absolute include paths.
- [ ] No CLI progress output or folding behavior is added in this slice.

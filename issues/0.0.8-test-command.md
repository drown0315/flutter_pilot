# 0.0.8 Test Command Issue Breakdown

Parent: https://github.com/drown0315/flutter_pilot/issues/81

Publishing status: Published to GitHub with the `ready-for-agent` label. Issues
82 through 87 are registered as sub-issues of the parent issue.

Historical note: physical iOS Recording Device discovery now exposes
helper-discovered AVFoundation devices only. `xctrace`-only devices are not
recordable and must not be treated as Recording Devices.

## 1. Store Device Video Recording Under Run Artifacts

GitHub: https://github.com/drown0315/flutter_pilot/issues/82

## What to build

Persist Scenario Recording video files inside the Scenario Run artifact bundle
instead of leaving them outside the run directory. A Device Video Recording is a
run-level artifact, not a Step artifact, so the saved video should live under
the run directory artifact area and report metadata should reference it with a
run-directory-relative path.

The completed slice should make recorded video artifacts portable with the rest
of the run directory and should keep the file name stable across devices.

## Acceptance criteria

- [ ] A recording-enabled Scenario Run writes the Device Video Recording under the run directory artifact area.
- [ ] The Device Video Recording uses a stable base name such as `device-video-recording` with the backend-native extension.
- [ ] Run report artifact metadata records the Device Video Recording path relative to the run directory.
- [ ] The Device Video Recording remains a run-level artifact and does not appear as a Step artifact.
- [ ] Artifact/report tests verify the stable video location and run-relative metadata.

## Blocked by

None - can start immediately

## 2. Add Target Device Resolution

GitHub: https://github.com/drown0315/flutter_pilot/issues/83

## What to build

Add Target Device resolution for the future `test` command. Flutter Pilot should
load Flutter Devices from `flutter devices --machine`, resolve a user-provided
Target Device selector by Flutter Device id, exact name, or unique id/name
prefix, and reject unsupported or ambiguous devices.

When Scenario Recording is required, the resolved Target Device must also pair
with a Recording Device by exact id or by a unique exact name match. If no
selector is provided and recording is required, Flutter Pilot should
auto-select only when exactly one supported Flutter Device has a paired
Recording Device.

## Acceptance criteria

- [ ] Target Device resolution returns `null` when no selector is provided and recording is not required.
- [ ] A non-empty selector resolves to exactly one supported Flutter Device by id, exact name, or unique id/name prefix.
- [ ] Empty, whitespace-only, missing, ambiguous, and unsupported Target Device selectors fail with exit-code-64 command errors.
- [ ] Recording-required resolution matches Recording Devices by resolved Flutter Device exact id or by unique exact name.
- [ ] Recording-required resolution auto-selects only when exactly one supported Flutter Device has a paired Recording Device.
- [ ] Recording-required resolution fails before app launch when there are zero or multiple recordable Target Devices.
- [ ] Error messages use Target Device terminology and list useful candidate devices when selection is ambiguous.

## Blocked by

- Physical iOS Recording Device discovery must expose only helper-discovered AVFoundation devices.

## 3. Launch Target App Package From Test Command Infrastructure

GitHub: https://github.com/drown0315/flutter_pilot/issues/84

## What to build

Add a Target App launcher that starts the current Target App Package with
`flutter run --machine`, extracts the Runtime Target URI from the
`app.debugPort.wsUri` machine event, hides raw machine stdout during normal
startup, buffers the last 40 stderr lines for launch diagnostics, and cleans up
the launched Flutter process.

The launcher should be a reusable deep module. It should not parse Scenarios,
resolve Target Devices, run Steps, or handle Scenario Recording directly.

## Acceptance criteria

- [ ] The launcher builds `flutter run --machine` commands using an optional Target Device id, optional Flutter flavor, and optional Flutter app entrypoint file.
- [ ] The launcher does not add explicit debug, pub, VM service port, or launch timeout options.
- [ ] The launcher extracts an absolute Runtime Target URI from supported `app.debugPort.wsUri` machine event shapes.
- [ ] Raw machine stdout is consumed but not printed during normal startup.
- [ ] If Flutter exits before a Runtime Target URI is available, launch fails without producing a Scenario Run and reports the last 40 stderr lines when present.
- [ ] The launcher supports cleanup by first sending Flutter's quit command and falling back to terminating the process when needed.
- [ ] Launcher tests use fake process streams and do not require a live Flutter app.

## Blocked by

None - can start immediately

## 4. Replace Run With Test Command

GitHub: https://github.com/drown0315/flutter_pilot/issues/85

## What to build

Remove the `run` command and add `test <scenario.yaml>` as the only CLI command
that executes a Scenario. `test` should parse and validate exactly one Scenario
file before launching Flutter, resolve Target Device rules when needed, launch
the current Target App Package, create the internal Runtime Target from the
discovered VM service URI, and then reuse the existing ScenarioRunner.

The command should preserve existing Scenario debugging controls under the new
execution entrypoint.

## Acceptance criteria

- [ ] `flutter_pilot run ...` is no longer registered as a command.
- [ ] `flutter_pilot test <scenario.yaml>` is registered as a public command and appears in help.
- [ ] `test` accepts `--device` / `-d`, `--flavor`, `--target` / `-t`, `--until`, repeated `--print`, and `--json`.
- [ ] `test` requires exactly one Scenario file and does not support directory, glob, or no-argument suite execution.
- [ ] Scenario validation failures happen before Flutter app launch.
- [ ] `--print` remains valid only with `--until`, and `--json` remains scoped to printed diagnostics.
- [ ] `test` prepares Scenario Recording before app launch when the backend requires it, then starts the saved segment only after the Runtime Target URI is available and before the first Scenario Step executes.
- [ ] `test` stops Scenario Recording before cleaning up the launched Flutter app.
- [ ] `test` cleans up the launched Flutter process when the Scenario succeeds, fails, or is interrupted.
- [ ] `validate`, `doctor`, `init`, `report`, and `diff` remain available and do not execute Scenarios.

## Blocked by

- 2. Add Target Device Resolution
- 3. Launch Target App Package From Test Command Infrastructure

## 5. Record Target Device Metadata In Run Reports

GitHub: https://github.com/drown0315/flutter_pilot/issues/86

## What to build

Include Target Device metadata in run reports for `test` runs that resolve a
Target Device. The metadata should describe the Flutter Device selected for the
run without turning Recording Device backend details into the Target Device
model.

This slice should preserve existing run directory naming while making run
artifacts explain which device produced them.

## Acceptance criteria

- [ ] Run reports for `test` runs with a resolved Target Device include Target Device id, name, Flutter target platform, emulator flag, and sdk string.
- [ ] Run reports do not include Flutter Device capabilities or `isSupported` in Target Device metadata.
- [ ] Run reports do not expose `screen_recorder` Recording Device objects as Target Device metadata.
- [ ] Run directory names do not include Target Device id or name.
- [ ] `test` output shows the resolved Target Device when one is selected.
- [ ] Report tests cover Target Device metadata presence and absence.

## Blocked by

- 2. Add Target Device Resolution
- 4. Replace Run With Test Command

## 6. Update Test Command Documentation

GitHub: https://github.com/drown0315/flutter_pilot/issues/87

## What to build

Update user-facing documentation and command references so `test` is described
as Flutter Pilot's only Scenario execution command. Documentation should remove
the old `run` flow, explain that `test` launches the current Target App Package,
describe supported app launch options, and document the Target Device rules for
Scenario Recording.

The completed slice should leave README, Chinese README, Scenario YAML docs,
and product docs with one consistent execution story.

## Acceptance criteria

- [ ] Documentation describes `flutter_pilot test <scenario.yaml>` as the Scenario execution command.
- [ ] Documentation no longer instructs users to run Scenarios by passing a VM service URI to `run`.
- [ ] Documentation explains `--device` / `-d`, `--flavor`, and `--target` / `-t`.
- [ ] Documentation explains that Scenario Recording requires the Target Device to pair with a Recording Device by exact id or unique exact name.
- [ ] Documentation explains that `validate` remains schema-only and that `doctor` / `init` remain explicit setup commands.
- [ ] Documentation states that full-suite Scenario discovery, arbitrary Flutter argument passthrough, and user-supplied VM service URI execution are out of scope for this slice.
- [ ] Documentation uses Scenario, Target App Package, Runtime Target, Target Device, Recording Device, and Scenario Recording terminology consistently.

## Blocked by

- 1. Store Device Video Recording Under Run Artifacts
- 4. Replace Run With Test Command
- 5. Record Target Device Metadata In Run Reports

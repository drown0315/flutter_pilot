## 1. Add minimal Run Diff command for Step outcomes

Type: AFK

## What to build

Add the first end-to-end `diff <before-run> <after-run>` path. The command
should compare two Scenario Runs from their run directories and print a
human-readable Run Diff focused on Step outcomes. This slice establishes the
Run Diff engine, report loading, CLI command, human-readable renderer, warnings
for different Scenario names, and exit-code behavior.

The Step comparison should align Steps by Step Label first and fall back to
Step index for unlabeled Steps. It should report resolved Steps, Step
Regressions, missing labeled Steps as Regressions, missing unlabeled Steps as
non-Regression missing findings, added Steps, and action changes.

## Acceptance criteria

- [ ] `flutter_pilot diff <before-run> <after-run>` reads both run directories
      and prints a human-readable Run Diff.
- [ ] The command returns exit code `0` when a Run Diff is successfully
      generated, even when Regressions are present.
- [ ] Usage errors return the existing CLI usage error behavior.
- [ ] Missing run directories, missing `run_report.json`, malformed JSON, or
      unsupported report shapes return a non-zero execution error with a clear
      message.
- [ ] Different Scenario names produce a warning and do not fail the command.
- [ ] Step alignment uses Step Label first and falls back to Step index for
      unlabeled Steps.
- [ ] Failed-to-passed Steps are reported as resolved Steps.
- [ ] Passed-to-failed Steps are reported as Regressions.
- [ ] Labeled Steps missing from the after run are reported as Regressions.
- [ ] Unlabeled Steps missing from the after run are reported as missing but
      are not automatically Regressions.
- [ ] Steps added in the after run are reported without being marked as
      Regressions.
- [ ] Aligned Steps with changed actions are reported as action changes.
- [ ] Tests cover unchanged runs, resolved Steps, Step Regressions, missing
      labeled Steps, missing unlabeled Steps, added Steps, action changes,
      scenario-name warnings, malformed reports, and CLI exit codes.

## Blocked by

None - can start immediately

## 2. Add machine-readable Run Diff JSON output

Type: AFK

## What to build

Add `diff --json` for machine-readable Run Diff output. JSON should be more
complete than the human-readable summary and expose a stable schema for agents,
CI, and downstream tooling. This slice should include detailed before and after
Step identity, warnings, Regression information, resolved Step information, and
an overall Run Diff outcome.

The outcome should be one of `unchanged`, `improved`, `regressed`, or
`changed`, with priority `regressed`, then `improved`, then `changed`, then
`unchanged`.

## Acceptance criteria

- [ ] `flutter_pilot diff <before-run> <after-run> --json` prints a stable JSON
      object to stdout.
- [ ] JSON output includes the before and after run directory values.
- [ ] JSON output includes warnings such as different Scenario names.
- [ ] JSON output includes Step findings with before and after index, label,
      action, status, and failure reason when present.
- [ ] JSON output identifies resolved Steps separately from Regressions.
- [ ] JSON output includes added, missing, action-changed, and status-changed
      Step findings.
- [ ] JSON output includes `outcome` with `unchanged`, `improved`,
      `regressed`, or `changed`.
- [ ] The outcome is `regressed` when any Regression exists.
- [ ] The outcome is `improved` when there are resolved Steps and no
      Regressions or neutral-only changes.
- [ ] The outcome is `changed` when there are changes but no resolved Steps or
      Regressions.
- [ ] The outcome is `unchanged` when there are no reported changes or
      warnings.
- [ ] Tests cover JSON schema shape, detailed Step identity, warnings, and all
      outcome values.

## Blocked by

- Add minimal Run Diff command for Step outcomes

## 3. Add visible text and runtime failure diffing

Type: AFK

## What to build

Extend Run Diff beyond Step outcomes by comparing run-level visible text and
runtime failures. Runtime failures should come from diagnostic summaries when
present, or from Logs artifacts reduced through the existing diagnostic reducer
when summaries are absent. Visible text should come from diagnostic summaries
when present, or from Snapshot artifacts reduced through the diagnostic reducer
when summaries are absent.

Resolved errors should mean runtime failures that were present before and
absent after. New errors should mean runtime failures that were absent before
and present after, and new errors should count as Regressions. Step
`failureReason` should remain part of Step diff only and must not be treated as
a runtime failure.

## Acceptance criteria

- [ ] Human-readable output reports visible text added and removed between
      runs.
- [ ] JSON output reports visible text added and removed between runs.
- [ ] Human-readable output reports resolved runtime failures.
- [ ] JSON output reports resolved runtime failures.
- [ ] Human-readable output reports new runtime failures as Regressions.
- [ ] JSON output reports new runtime failures as Regressions.
- [ ] Runtime failure extraction uses top-level diagnostic summaries when
      available.
- [ ] Runtime failure extraction falls back to Logs artifacts when summaries are
      absent.
- [ ] Visible text extraction uses top-level diagnostic summaries when
      available.
- [ ] Visible text extraction falls back to Snapshot artifacts when summaries
      are absent.
- [ ] Missing referenced Snapshot or Logs artifacts produce warnings rather than
      failing the whole Run Diff.
- [ ] Step `failureReason` does not appear in resolved errors or new errors.
- [ ] Tests cover visible text from summaries, visible text from Snapshot
      artifacts, resolved runtime failures, new runtime failures, missing
      artifacts, and the Step `failureReason` boundary.

## Blocked by

- Add machine-readable Run Diff JSON output

## 4. Add Screenshot artifact difference reporting

Type: AFK

## What to build

Add Screenshot artifact difference reporting to Run Diff. The first version
should compare Screenshot artifacts by presence and file hash only. It should
not perform pixel-level visual diffing, heatmap generation, thresholding, or
perceptual image comparison.

Screenshot changes should be reported as useful review signals but should not
automatically count as Regressions. JSON output should include enough metadata
for downstream tools to locate and inspect both sides of the change.

## Acceptance criteria

- [ ] Human-readable output reports Screenshot artifacts added in the after run.
- [ ] Human-readable output reports Screenshot artifacts missing from the after
      run.
- [ ] Human-readable output reports Screenshot artifacts whose file hash changed.
- [ ] JSON output reports Screenshot changes with Step key, change type,
      before Step identity, after Step identity, artifact paths, and hashes.
- [ ] Screenshot hash changes do not create Regressions by themselves.
- [ ] Missing referenced Screenshot files produce warnings rather than failing
      the whole Run Diff.
- [ ] The implementation does not perform pixel-level comparison or generate
      heatmaps.
- [ ] Tests cover added screenshots, missing screenshots, changed screenshots,
      missing screenshot artifacts, JSON metadata, and the non-Regression
      treatment of screenshot changes.

## Blocked by

- Add machine-readable Run Diff JSON output

## 5. Polish Run Diff documentation and acceptance fixtures

Type: AFK

## What to build

Finalize the Run Diff slice with user-facing documentation and comprehensive
acceptance fixtures. Documentation should explain the command, human-readable
output, `--json` output, exit-code semantics, outcome values, warnings,
Regression rules, artifact/hash-only Screenshot comparison, and the first
version boundaries.

Acceptance fixtures should make it easy for future agents and maintainers to
verify unchanged, improved, regressed, changed, malformed, and partial-artifact
Run Diff behavior without a live Flutter app.

## Acceptance criteria

- [ ] User-facing documentation includes `diff <before-run> <after-run>`.
- [ ] Documentation includes a human-readable output example.
- [ ] Documentation includes a `--json` output example.
- [ ] Documentation explains that successful Run Diff generation returns exit
      code `0` even when Regressions are present.
- [ ] Documentation explains `unchanged`, `improved`, `regressed`, and
      `changed` outcomes.
- [ ] Documentation explains Step alignment by Step Label with index fallback.
- [ ] Documentation explains that runtime failures are separate from Step
      failure reasons.
- [ ] Documentation explains that Screenshot comparison is artifact/hash-only.
- [ ] Documentation explicitly lists out-of-scope behavior: pixel-level visual
      diffing, heatmaps, Widget Tree diffing, interactive widget summary
      diffing, automatic output files, and CI gating.
- [ ] Fixtures or tests cover unchanged, improved, regressed, changed,
      malformed, and partial-artifact runs.

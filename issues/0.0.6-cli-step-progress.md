# 0.0.6 CLI Step Progress Issue Breakdown

Parent: https://github.com/drown0315/flutter_pilot/issues/66

Publishing status: Published to GitHub with the `ready-for-agent` label.

## 1. Add Plain Step Progress For Successful Runs

GitHub: https://github.com/drown0315/flutter_pilot/issues/74

## Parent

https://github.com/drown0315/flutter_pilot/issues/66

## What to build

Add runner progress events, a deterministic plain-text progress renderer, CLI
stderr wiring, and a test-only fake runtime path so `flutter_pilot test` can
show live progress for successful Steps without changing Scenario semantics or
artifact formats. The slice should cover Scenario start information, Step
number and label display, success status, Step duration, stdout/stderr
separation, and `test --json` progress suppression at the command boundary.

## Acceptance criteria

- [ ] `flutter_pilot test` emits Step started and Step finished progress for successful Steps.
- [ ] Progress output includes Scenario name, total Step count, Step number, action, and Step Label or `-` when unlabeled.
- [ ] Successful Step output includes a clear success status and elapsed duration.
- [ ] Human-readable progress is written to stderr, while final report paths remain on stdout.
- [ ] `test --json` suppresses progress output.
- [ ] CLI subprocess tests can exercise the success path using a test-only fake Runtime Adapter selection.

## Blocked by

None - can start immediately

## 2. Render Failed And Unlabeled Step Progress

GitHub: https://github.com/drown0315/flutter_pilot/issues/75

## Parent

https://github.com/drown0315/flutter_pilot/issues/66

## What to build

Extend the Step progress path to cover failed Steps and unlabeled Steps. The
completed slice should make failures easy to identify from terminal output
while preserving full failure details in the run report and artifacts.
Unlabeled Steps should remain readable through the same progress layout used by
labeled Steps.

## Acceptance criteria

- [ ] Failed Step progress includes Step number, action, Step Label or `-`, failed status, elapsed duration, and a concise failure reason.
- [ ] Failure summaries collapse newlines to spaces and truncate overly long messages in progress output only.
- [ ] Full failure reasons remain unchanged in `run_report.json` and diagnostic artifacts.
- [ ] Unlabeled Steps render with `-` in the label column.
- [ ] Action and label columns remain aligned in deterministic plain-text output.
- [ ] Tests cover failed Step progress, unlabeled Step progress, summary truncation, and preservation of full failure details.

## Blocked by

- https://github.com/drown0315/flutter_pilot/issues/74

## 3. Handle Until Json And Final Human Summaries

GitHub: https://github.com/drown0315/flutter_pilot/issues/76

## Parent

https://github.com/drown0315/flutter_pilot/issues/66

## What to build

Complete the CLI progress contract for `--until`, `--json`, and final human
summaries. The slice should make stopped runs clearly distinguishable from
failures, keep JSON output free of progress text, and preserve the existing
stdout artifact-path contract.

## Acceptance criteria

- [ ] Successful `--until` runs print a clear stopped summary that references the selected stop point.
- [ ] Successful complete runs print `Run passed.` in the human-readable summary.
- [ ] Failed runs print `Run failed at ...` in the human-readable summary.
- [ ] Only executed Steps produce progress output when `--until` is used.
- [ ] `test --json` prints no Step progress.
- [ ] Existing stdout lines for `Run report:` and `HTML report:` remain unchanged.
- [ ] Tests cover stopped runs, complete runs, failed runs, and `test --json` suppression.

## Blocked by

- https://github.com/drown0315/flutter_pilot/issues/74
- https://github.com/drown0315/flutter_pilot/issues/75

## 4. Add Interactive Terminal Styling And In Place Refresh

GitHub: https://github.com/drown0315/flutter_pilot/issues/77

## Parent

https://github.com/drown0315/flutter_pilot/issues/66

## What to build

Add the internal terminal styling module used by the CLI progress renderer. The
slice should provide ANSI color, bold and dim text, emoji capability, ANSI
stripping, TTY-aware enablement, and in-place status refresh for interactive
terminals while keeping non-TTY output plain and deterministic.

## Acceptance criteria

- [ ] Interactive terminals render progress with color, emoji, and in-place status refresh.
- [ ] Non-interactive output remains plain text with no ANSI codes or emoji.
- [ ] The styling module can color, bold, dim, and strip text.
- [ ] TTY capability detection is isolated from progress wording and run semantics.
- [ ] Tests cover enabled styling, disabled styling, ANSI stripping, and TTY-aware behavior.

## 1. Discover Project Scenarios

Issue: https://github.com/drown0315/flutter_pilot/issues/99

## Parent

https://github.com/drown0315/flutter_pilot/issues/98

## What to build

Add Project Scenario discovery for Project Run mode. The feature should discover
Entry Scenario files from the default Pilot Directory when `test` has no
positional argument, and from an explicit directory when the user passes one.
Discovery must recurse through `.yaml` and `.yml` files, run only files with
top-level `scenario:` metadata, treat metadata-free YAML files as Step Library
candidates, return deterministic path order, and fail clearly when no Project
Scenarios are found.

## Acceptance criteria

- [x] Discovery uses `pilot/` as the default Pilot Directory for no-argument `test`.
- [x] Discovery accepts an explicit directory root for focused Project Runs.
- [x] Discovery recursively scans `.yaml` and `.yml` files.
- [x] Discovery includes only files with top-level `scenario:` metadata.
- [x] Metadata-free YAML files are ignored as runnable Entry Scenarios.
- [x] Discovered Project Scenarios are ordered lexicographically by POSIX-style path relative to the discovery root.
- [x] Empty discovery returns a usage-level error before app launch.
- [x] Tests cover default root, explicit root, nested files, extension handling, Step Library exclusion, ordering, invalid YAML, validation errors, and empty discovery.

## Progress

Done in this repository:

- Added `ProjectScenarioDiscovery` and `ProjectScenarioFile`.
- Added a usage-level `ProjectScenarioDiscoveryException` for missing roots and empty discovery.
- Added `test/project_scenario_discovery_test.dart` coverage for the acceptance criteria above.

Validation:

- `dart format .`
- `dart analyze`
- `dart test`
- `dart test test/project_scenario_discovery_test.dart`

## Blocked by

None - can start immediately.

## 2. Select Test Mode From CLI Inputs

Issue: https://github.com/drown0315/flutter_pilot/issues/101

## Parent

https://github.com/drown0315/flutter_pilot/issues/98

## What to build

Update `test` command argument handling so file, directory, and no-argument
invocations select the correct execution mode. Single-file mode should preserve
existing Entry Scenario behavior and debugging options. Project Run mode should
reject ambiguous or unsupported inputs, reject `--until` and `--print`, and
prepare validated Project Scenarios before app launch.

## Acceptance criteria

- [x] `flutter_pilot test <scenario.yaml>` continues to run a single Entry Scenario.
- [x] `flutter_pilot test` selects Project Run mode from `pilot/`.
- [x] `flutter_pilot test <directory>` selects Project Run mode from that directory.
- [x] Multiple positional inputs are rejected as usage errors.
- [x] Glob-style inputs are not introduced as supported behavior.
- [x] Single-file mode continues to allow omitted Scenario metadata.
- [x] Project Run mode rejects `--until`.
- [x] Project Run mode rejects `--print`.
- [x] CLI tests cover file, directory, no-argument, multiple-input, empty-discovery, and illegal debug-option behavior.

## Progress

Done in this repository:

- Updated `test` mode selection for no argument, one file, and one directory.
- Added a Project Run executor boundary used after discovery and validation.
- Preserved single Entry Scenario mode, including metadata-free file inputs.
- Rejected multiple inputs, unsupported glob-like paths, empty discovery, `--until`, and `--print` before Project Run execution.

## 3. Group Project Run Artifacts

Issue: https://github.com/drown0315/flutter_pilot/issues/102

## Parent

https://github.com/drown0315/flutter_pilot/issues/98

## What to build

Add Project Run artifact grouping without changing existing single-Scenario run
artifacts. A Project Run should create one batch-level run directory under
`.runs/`, place each Scenario Run in its own child directory, and write a
machine-readable `project_run_report.json` at the batch root. Existing Scenario
Run reports and timelines should remain usable inside child directories.

## Acceptance criteria

- [x] Project Run creates one batch-level run directory under `.runs/`.
- [x] Each Scenario Run in the batch writes to a child run directory.
- [x] Child run directories keep the existing Scenario Run layout.
- [x] The batch root contains `project_run_report.json`.
- [x] `project_run_report.json` records discovery root, Scenario file paths, execution order, per-Scenario status, report paths, Project Run status, duration, command inputs, and environment-level failure details when present.
- [x] The first version does not generate a batch-level HTML report.
- [x] Existing `report` and `diff` behavior for Scenario Run directories is preserved.
- [x] Tests cover directory creation, relative report paths, child Scenario Run layout, all-passed reports, partially failed reports, and environment-level failure reports.

## Progress

Done in this repository:

- Added `ProjectRunArtifactWriter` for batch-level Project Run directories.
- Added child Scenario Run creation that preserves existing Scenario Run layout.
- Added `project_run_report.json` writing at the batch root.
- Added Project Run report models and JSON serialization for passed, partially failed, and environment-failed batches.
- Added batch-root-relative child report path helpers.

Validation:

- `dart test test/artifact_store_test.dart`
- `dart test test/project_run_report_test.dart`

## 4. Hot Restart Launched Target App

Issue: https://github.com/drown0315/flutter_pilot/issues/100

## Parent

https://github.com/drown0315/flutter_pilot/issues/98

## What to build

Expose hot restart as an operation on the launched Target App process. The
Target App Launcher should own the Flutter process communication needed to
request a hot restart and report success or failure through a small testable
interface. This slice should not orchestrate a full Project Run; it only makes
the launched app reusable between Scenarios.

## Acceptance criteria

- [x] A launched Target App handle exposes a hot restart operation.
- [x] Hot restart uses Flutter process control rather than Scenario semantics.
- [x] Hot restart success is observable by callers.
- [x] Hot restart failure returns a structured launch/process-level failure.
- [x] Existing cleanup behavior remains intact.
- [x] Fake process tests cover hot restart success.
- [x] Fake process tests cover hot restart failure.
- [x] Fake process tests cover cleanup after hot restart activity.

## Progress

Done in this repository:

- Added `TargetAppLaunch.hotRestart()`.
- Hot restart writes Flutter CLI's restart control command to the launched process.
- Hot restart write failures are reported as `TargetAppLaunchException`.
- Added fake process tests for restart success, restart failure, and cleanup after restart activity.

Validation:

- `dart test test/target_app_launcher_test.dart`

## Blocked by

None - can start immediately.

## 5. Execute Project Runs With App Reuse

Issue: https://github.com/drown0315/flutter_pilot/issues/103

## Parent

https://github.com/drown0315/flutter_pilot/issues/98

## What to build

Implement Project Run orchestration across discovery, launch, hot restart,
Scenario execution, artifacts, and final status. A Project Run should launch the
Target App Package once, run the first Scenario directly, hot restart before
later Scenarios, continue after Scenario execution failures, stop on
environment-level failures, and produce the correct Project Run status and exit
code.

## Acceptance criteria

- [ ] Project Run launches the Target App Package once.
- [ ] The first Scenario runs without a preliminary hot restart.
- [ ] Later Scenarios run after a hot restart.
- [ ] Flutter Pilot hot restarts after a failed Scenario before continuing.
- [ ] Scenario execution failure records that Scenario's artifacts and does not stop later Scenarios.
- [ ] Launch failure, Target Device resolution failure, Project Scenario validation failure, discovery failure, and hot restart failure stop the Project Run.
- [ ] Project Run exits `0` only when every Scenario passes.
- [ ] Project Run exits `1` when one or more Scenario Runs fail.
- [ ] Tests use fake launch, fake hot restart, and fake runner boundaries.
- [ ] Tests cover launch reuse, continuation after Scenario failure, stop on hot restart failure, Project Run report status, and final exit code.

## 7. Integrate Project Run Device And Recording Rules

Issue: https://github.com/drown0315/flutter_pilot/issues/105

## Parent

https://github.com/drown0315/flutter_pilot/issues/98

## What to build

Ensure Project Run mode uses the existing launch option, Target Device, and
Scenario Recording rules coherently across a batch. `--device`, `--flavor`, and
`--target` should apply to the single launched Target App Package. Target Device
resolution should happen once before launch. Scenario Recording intent should
remain Scenario-driven and never silently degrade.

## Acceptance criteria

- [ ] `--device` applies to Project Run mode.
- [ ] `--flavor` applies to Project Run mode.
- [ ] `--target` applies to Project Run mode.
- [ ] Target Device resolution happens once before Project Run launch.
- [ ] Recording-required Project Scenarios use the existing Target Device and Recording Device matching rules.
- [ ] Recording-enabled Project Scenarios never silently run without recording.
- [ ] No Project Run CLI flag is added to enable or disable Scenario Recording.
- [ ] Tests cover explicit Target Device selection, recording-required auto-selection, ambiguous recording-required selection, and launch option propagation.

- https://github.com/drown0315/flutter_pilot/issues/103

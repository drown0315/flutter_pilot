## 1. Add Plain Target App Launch Progress

GitHub: https://github.com/drown0315/flutter_pilot/issues/91

## Parent

https://github.com/drown0315/flutter_pilot/issues/90

## What to build

Add the first end-to-end Target App Launch Progress path for `flutter_pilot
test`. Human-readable runs should print deterministic launch status to stderr
while the Target App Package is being launched and Flutter Pilot is waiting for
the Runtime Target. The completed slice should show that app launch is active,
include elapsed launch time, leave a successful launch summary, then hand off to
existing Step progress without changing stdout report paths.

This slice should keep the experience intentionally plain and deterministic. It
does not need interactive spinner behavior, device selection details, stage
detection beyond a reliable waiting state, or heartbeat output.

## Acceptance criteria

- [x] Non-JSON `flutter_pilot test` prints Target App Launch Progress before Step progress begins.
- [x] Launch progress is written to stderr while `Run report:` and `HTML report:` remain on stdout.
- [x] Launch progress includes a clear launch-in-progress status and elapsed launch time.
- [x] Successful launch leaves a summary line such as `Target App launched in 38s`.
- [x] Step progress begins after the launch success summary as a separate block.
- [x] `test --json` suppresses Target App Launch Progress.
- [x] Renderer or command tests cover plain launch progress, stdout/stderr separation, success summary, and JSON suppression without launching a real Flutter app.

Progress:

- Implemented `TargetAppLaunchProgressEvent` events and a deterministic
  `TargetAppLaunchProgressRenderer`.
- Wired non-JSON `flutter_pilot test` to stderr launch progress and kept JSON
  output suppressed.
- Added command/executor tests that verify launch progress wiring, success
  summary ordering before Step progress, and JSON suppression without launching
  a real Flutter app.

## Blocked by

None - can start immediately

## 2. Show Launch Choices And Device Selection Reason

GitHub: https://github.com/drown0315/flutter_pilot/issues/92

## Parent

https://github.com/drown0315/flutter_pilot/issues/90

## What to build

Extend Target App Launch Progress to show the launch choices that explain what
Flutter Pilot is starting. The launch panel should show the resolved Target
Device when one is known, explain whether it came from the user-provided
`--device` value or Scenario Recording auto-selection, and show flavor and
entrypoint when provided.

When Flutter Pilot intentionally lets Flutter choose its default device, the
panel should say `Target Device: Flutter default` rather than inventing a device
name. This preserves Flutter default behavior while making the device-selection
boundary visible.

## Acceptance criteria

- [x] Launch progress shows the resolved Target Device for explicit `--device` selections.
- [x] Launch progress shows the original device selector as the selection reason for explicit `--device` selections.
- [x] Launch progress shows the resolved Target Device for Scenario Recording auto-selection.
- [x] Launch progress shows `Selection: auto-selected for recording` for recording-driven auto-selection.
- [x] Launch progress shows `Target Device: Flutter default` when no Target Device was resolved and Flutter chooses the device.
- [x] Launch progress shows flavor when `--flavor` is provided.
- [x] Launch progress shows entrypoint when `--target` is provided.
- [x] Omitted Target Device metadata, flavor, and entrypoint do not produce noisy placeholder lines beyond the intentional `Flutter default` device label.
- [x] Tests cover explicit device selection, recording auto-selection, Flutter default, flavor display, and entrypoint display.

Progress:

- Added `TargetAppLaunchChoices` and Target Device selection reasons for
  explicit `--device` and Scenario Recording auto-selection.
- Rendered resolved Target Device details, Flutter default selection, flavor,
  and entrypoint in Target App Launch Progress.
- Wired default `test` execution to pass resolved Target Device metadata and
  launch options into launch progress events.

## 3. Add Launch Failure Progress Output

GitHub: https://github.com/drown0315/flutter_pilot/issues/93

## Parent

https://github.com/drown0315/flutter_pilot/issues/90

## What to build

Make Target App Launch Progress fail clearly when the Target App Package cannot
provide a Runtime Target. Startup failures should show a concise failure
summary, elapsed launch time, and the buffered Flutter stderr tail. Default
output should remain bounded and high signal rather than streaming full Flutter
build logs.

The completed slice should preserve the existing launch failure behavior while
making it visible in the same Target App Launch Progress surface users watched
during startup.

## Acceptance criteria

- [x] Launch failure output includes a clear Target App launch failure summary.
- [x] Launch failure output includes elapsed launch time.
- [x] Launch failure output includes the buffered Flutter stderr tail when available.
- [x] Launch failure output remains bounded to the existing stderr tail behavior rather than printing full Flutter logs.
- [x] Launch failure output is written to stderr.
- [x] `test --json` suppresses launch progress output while preserving command failure behavior.
- [x] Tests cover Flutter exiting before Runtime Target availability, stderr tail display, bounded output, and JSON suppression.

Progress:

- Added `TargetAppLaunchFailedEvent` with failure time, message, launch
  choices, and buffered Flutter stderr tail.
- Rendered launch failure summaries with elapsed time and the existing bounded
  stderr tail.
- Wired default `test` execution to emit failure progress before preserving the
  existing `TestCommandException` failure path.

## 4. Add Interactive Launch Loading Panel

GitHub: https://github.com/drown0315/flutter_pilot/issues/94

## Parent

https://github.com/drown0315/flutter_pilot/issues/90

## What to build

Add the interactive Target App Launch Progress experience for real terminals.
Interactive output should render a compact loading panel with an indeterminate
spinner, elapsed time, launch choices, and reliable high-level launch stage. The
panel should refresh in place and leave the successful launch summary behind
before Step progress starts.

The implementation should not show fake percentages and should not guess build
or install stages without reliable signals. When finer-grained stage detection
is not reliable, the panel should honestly show that Flutter Pilot is waiting
for the Runtime Target.

## Acceptance criteria

- [x] Interactive terminals render Target App Launch Progress as a compact in-place loading panel.
- [x] The interactive panel includes an indeterminate spinner and elapsed time.
- [x] The interactive panel includes launch choices from the existing launch progress data.
- [x] The interactive panel leaves a final success summary before Step progress starts.
- [x] Non-interactive output remains deterministic plain text without ANSI animation.
- [x] No percentage progress is shown.
- [x] High-level stages are shown only when supported by reliable launch information; otherwise the panel falls back to waiting for Runtime Target.
- [x] Renderer tests cover interactive refresh behavior, spinner-compatible output, plain non-interactive fallback, and success handoff.

Progress:

- Added interactive Target App Launch Progress rendering with in-place ANSI
  refresh for real terminals.
- Reused launch choices in the interactive panel and used the honest
  `Waiting for Runtime Target` stage without percentage progress.
- Left success and failure summaries as deterministic final lines after
  clearing the interactive panel.

## 5. Add Non-Interactive Launch Heartbeats

GitHub: https://github.com/drown0315/flutter_pilot/issues/95

## Parent

https://github.com/drown0315/flutter_pilot/issues/90

## What to build

Add periodic heartbeat output for non-interactive Target App Launch Progress.
When app launch takes a long time in CI or redirected output, Flutter Pilot
should print deterministic status lines at a low cadence so logs show the
command is still active without being flooded.

The initial cadence should be ten seconds. Heartbeats should use the same launch
status wording as plain Target App Launch Progress and should stop once launch
succeeds or fails.

## Acceptance criteria

- [ ] Non-interactive launch progress prints heartbeat status while waiting for the Runtime Target.
- [ ] The initial heartbeat cadence is ten seconds.
- [ ] Heartbeat lines include elapsed launch time.
- [ ] Heartbeat output stops after launch success or launch failure.
- [ ] Heartbeat output is written to stderr.
- [ ] Interactive terminal output does not use the non-interactive heartbeat line behavior.
- [ ] Tests use a fake clock or injected ticker to verify heartbeat cadence without slow real-time waits.

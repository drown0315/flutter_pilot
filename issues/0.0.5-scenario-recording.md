# 0.0.5 Scenario Recording Issue Breakdown

Publishing status: Published to GitHub with the `ready-for-agent` label.

## 1. Add Scenario Recording DSL Validation

GitHub: https://github.com/drown0315/flutter_pilot/issues/69

## What to build

Add Scenario-level recording support to the strict Scenario DSL. Recording is a
Scenario metadata option under `scenario.recording`, not a Step Action and not
an extension of Capture Action. The first slice should carry recording through
the full validation path: YAML parsing, typed Scenario model, and CLI
`validate`.

The accepted forms are:

- `scenario.recording: {}`
- `scenario.recording.enabled: true`
- `scenario.recording.enabled: false`

This slice should normalize `scenario.recording: {}` to enabled recording,
preserve omitted recording metadata as missing typed state, and reject invalid
forms such as boolean shorthand or unknown recording fields.

## Acceptance criteria

- [ ] The typed Scenario model exposes optional Scenario Recording metadata rather than a Step Action or a flat boolean.
- [ ] `flutter_pilot validate` accepts `scenario.recording: {}` and `scenario.recording.enabled: true/false`.
- [ ] `flutter_pilot validate` rejects non-map recording values and unknown recording fields with structured validation paths.
- [ ] Parser and CLI validation tests cover omitted recording metadata, normalized default enablement, explicit disablement, and invalid recording schema.

## Blocked by

None - can start immediately

## 2. Start Scenario Recording Before Step Execution

GitHub: https://github.com/drown0315/flutter_pilot/issues/70

## What to build

Integrate Scenario Recording into Scenario Run lifecycle through a narrow,
fakeable recording boundary. When Scenario Recording is enabled, Flutter Pilot
should create a Recording Session before executing the first Step and stop the
run immediately if recording startup fails.

This slice is about runner behavior rather than final artifact persistence. It
should prove that recording belongs to run lifecycle, not step lifecycle, and
that recording-disabled runs behave exactly as they did before.

## Acceptance criteria

- [ ] A recording-enabled Scenario Run attempts to start a Recording Session before any Step executes.
- [ ] When recording startup fails, the Scenario Run fails before executing any Step and reports a clear run-level failure reason.
- [ ] Scenario Runs with omitted recording metadata or `enabled: false` do not attempt to start recording.
- [ ] Runner tests cover enabled, omitted, disabled, and startup-failure recording behavior through a fakeable recording boundary.

## Blocked by

- 1. Add Scenario Recording DSL Validation

## 3. Save Device Video Recording As Run Artifact

GitHub: https://github.com/drown0315/flutter_pilot/issues/71

## What to build

Persist the final Device Video Recording as a run-level artifact after a
recording-enabled Scenario Run completes. The recorded video belongs to the
Scenario Run, not to any individual Step, so artifact metadata and reports
should expose it at run scope.

This slice should complete the recording lifecycle by saving the final artifact
metadata in the same stable reporting model Flutter Pilot already uses for
other run artifacts.

## Acceptance criteria

- [ ] A successful recording-enabled Scenario Run stores the final Device Video Recording as a run-level artifact.
- [ ] The recorded video does not appear as a Step artifact.
- [ ] Run report metadata exposes the recorded video through stable run-level artifact records.
- [ ] Artifact and reporting tests verify the presence and scope of the Device Video Recording artifact.

## Blocked by

- 2. Start Scenario Recording Before Step Execution

## 4. Wire Flutter Pilot To `screen_recorder`

GitHub: https://github.com/drown0315/flutter_pilot/issues/72

## What to build

Connect Flutter Pilot's recording boundary to the real `screen_recorder`
package so that Scenario Recording can create and stop a real Recording
Session. The integration should keep Flutter Pilot's runner contract narrow and
must not move Device Video Recording into the Runtime Adapter.

This slice should normalize `screen_recorder` lifecycle and failure behavior
into Flutter Pilot run behavior while keeping most tests device-free through
fakes and contract boundaries.

## Acceptance criteria

- [ ] Flutter Pilot uses the real `screen_recorder` package behind its recording boundary when Scenario Recording is enabled.
- [ ] Recording start and stop behavior is normalized into Flutter Pilot run lifecycle without extending the Runtime Adapter contract.
- [ ] Recording failures from `screen_recorder` are surfaced as clear Flutter Pilot run-level failures.
- [ ] Integration-facing tests cover the recording boundary contract without requiring real recording devices.

## Blocked by

- 2. Start Scenario Recording Before Step Execution

## 5. Document Scenario Recording And CLI Behavior

GitHub: https://github.com/drown0315/flutter_pilot/issues/73

## What to build

Update Flutter Pilot documentation so Scenario Recording is described as an
optional Scenario-level recording feature rather than an out-of-scope or
step-level capability. The docs should explain the strict YAML shape, default
behavior, run-failure semantics, and run-level Device Video Recording artifact
scope.

This slice should leave future readers with one consistent story across glossary,
PRD, and user-facing documentation.

## Acceptance criteria

- [ ] Documentation explains that Scenario Recording lives under `scenario.recording` and is not a Step Action.
- [ ] Documentation explains the accepted YAML forms, omitted-recording behavior, and invalid shorthand behavior.
- [ ] Documentation explains that recording startup failure causes `run` to fail before Step execution.
- [ ] Documentation explains that Device Video Recording is a run-level artifact complementary to step-owned Screenshot artifacts.

## Blocked by

- 1. Add Scenario Recording DSL Validation
- 2. Start Scenario Recording Before Step Execution
- 3. Save Device Video Recording As Run Artifact
- 4. Wire Flutter Pilot To `screen_recorder`
